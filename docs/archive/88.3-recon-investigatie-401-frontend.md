## DE CE
=====================================================================

Promptul 88 (`sessionGuard`) a făcut ca orice rută `/api/`, `/flows/`, `/admin/` să poată întoarce
**401 mid-sesiune** (parolă resetată de admin, cont dezactivat, rol schimbat, org schimbat).
Frontend-ul trebuie să ducă utilizatorul la login, nu să-l lase pe un ecran mort.

Am pornit să scriu handler-ul global pe două premise. **Ambele s-au dovedit greșite:**

1. *„Nu există wrapper comun de fetch."* → **FALS.** `public/js/flow/flow.js:3` declară:
   `Depends on: _apiFetch (df-apifetch-shim.js)`. **Există deja un shim.** Nu l-am văzut pentru că
   am căutat `apiFetch`, nu `_apiFetch`.
2. *„`flow.html` e o pagină de semnare pentru semnatari externi."* → **FALS.** `flow.html` e
   **pagina de status a fluxului**, la care se ajunge din notificări (`notif-widget.js:311`,
   `sw.js:178` → `/flow.html?flow=…`, **fără token**). Pagina de semnare e **`semdoc-signer.html`**
   (`server/index.mjs:899`, `buildSignerLink`).

Și am descoperit ceva care schimbă designul: **`flow.js` folosește 401-ul ca SEMNAL DE CONTROL.**

```js
// public/js/flow/flow.js:42-48
async function apiFetchJson(url){
  let r = await _apiFetch(url);
  if((r.status===401 || r.status===403) && linkToken){        // ← 401 = „încearcă cu tokenul din link"
    const u = new URL(url, location.origin);
    if(!u.searchParams.get("token")) u.searchParams.set("token", linkToken);
    r = await _apiFetch(u.toString());                         // ← retry
  }
  ...
}
```

Un interceptor global care redirectează la `/login` pe primul 401 **ar fura acest semnal** și ar
rupe retry-ul. **Designul depinde complet de ce aflăm mai jos.**
