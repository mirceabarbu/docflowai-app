## EȘECUL 3 — `_checkLoginRate is not a function` → timeout 30s
=====================================================================

**Cauză dublă.**

### 3a. Bug de harness (cauza imediată)

`server/routes/auth.mjs:45` cheamă `await _checkLoginRate(req, email)`. `_checkLoginRate` e
**injectat** (`auth.mjs:34`), iar `createApp()` din testul DB **nu cheamă injectorul** ⇒ `null` ⇒
`TypeError` ⇒ cererea nu primește niciodată răspuns ⇒ timeout la 30s.

**Reparație:** în harness-ul testului, injectează funcțiile de rate-limit pentru login înainte de a
construi app-ul. Localizează injectorul (`grep -n "_checkLoginRate\s*=" server/routes/auth.mjs`
⇒ linia 34; găsește funcția exportată care îl setează) și cheam-o cu stub-uri no-op:

```js
// Rate-limiterul de login e injectat din index.mjs, care nu rulează în harness-ul de test.
// Fără injecție, auth.mjs:45 aruncă TypeError și cererea nu răspunde niciodată.
injectLoginRateLimiter(
  async () => ({ blocked: false }),   // check
  async () => {},                     // recordFail
  async () => {}                      // clear
);
```
(numele exact al exportului se citește din fișier — **nu-l ghici**)

> 💡 Verifică dacă `server/tests/db/helpers/` are deja un helper de app care face asta. Celelalte
> teste DB nu au lovit `/auth/login`, de aceea gaura n-a ieșit până acum. **Dacă adaugi injecția
> într-un helper partajat, fă-o aditiv — nu schimba comportamentul pentru testele existente.**

### 3b. Fragilitate de producție (expusă gratis de CI)

`auth.mjs:45` nu are **niciun null-guard**, spre deosebire de `checkTokenVersionValid`
(`middleware/auth.mjs:112`), care face `if (!_checkTokenVersion ...) return true`.

Dacă injectorul nu apucă să ruleze (ordine de import schimbată la o refactorizare),
**fiecare login dă 500**. Adaugă o gardă defensivă:

`old_str`:
```js
  const rateCheck = await _checkLoginRate(req, email);
```
`new_str`:
```js
  // SEC-87.1: rate-limiterul e injectat din index.mjs. Dacă injecția nu a rulat (ordine de import
  // schimbată la refactor), apelul arunca TypeError și TOATE loginurile dădeau 500 — o pană totală
  // de autentificare. Fail-open pe rate limiting (disponibilitate > fereastra de brute-force), dar
  // cu logger.error, ca deployment-ul rupt să fie imediat vizibil.
  if (typeof _checkLoginRate !== 'function') {
    logger.error('SEC: rate-limiterul de login NU este injectat — login permis FĂRĂ rate limiting. Verifică injecția din index.mjs.');
  }
  const rateCheck = typeof _checkLoginRate === 'function'
    ? await _checkLoginRate(req, email)
    : { blocked: false };
```

> ⚠️ **Aceasta e o decizie deliberată de fail-OPEN**, singura din tot sprintul. Motiv: un login
> complet blocat e mai rău decât o fereastră de brute-force care e oricum acoperită de
> `token_version`, TOTP și logarea erorii. **Dacă nu ești de acord, raportează în loc să aplici.**
>
> Aplică același tratament **doar** dacă `_recordLoginFail` / `_clearLoginRate` sunt apelate
> necondiționat pe aceeași cale — verifică și raportează.
