## PROBLEMA
=====================================================================

`server/routes/flows/crud.mjs:838-844`:

```js
router.get('/my-flows/:flowId/download', async (req, res) => {
  if (requireDb(res)) return;
  const qToken = req.query.token;
  let actor = null;
  if (qToken) { try { actor = jwt.verify(qToken, JWT_SECRET); } catch(e) {} }   // ← SESIUNE COMPLETĂ
  if (!actor) actor = requireAuth(req, res);
  if (!actor) return;
```

`qToken` este un **JWT complet de utilizator** (același `JWT_SECRET`, poartă `userId`, `email`,
`role`, `orgId`), validat **exclusiv pe semnătură** — fără `token_version`, fără `deleted_at`.

### Ocolirea

`sessionGuard.extractToken()` citește **doar** cookie-ul `auth_token` și
`Authorization: Bearer`. **Nu citește query string-ul.**

⇒ `GET /my-flows/123/download?token=<jwt>` **fără cookie**:
1. garda nu găsește token ⇒ `next()` (corect: nu rupem tokenurile funcționale de semnatar);
2. ruta verifică doar semnătura ⇒ servește PDF-ul semnat.

**Un cont dezactivat, un admin retrogradat sau o parolă resetată ocolesc întreaga gardă mutându-și
propriul JWT din cookie în URL.** Fără unelte, fără nimic. Anulează #88 și #88.1 pe această rută.

### Agravant

Un JWT de sesiune într-un query string ajunge, prin construcție, unde un cookie nu ajunge niciodată:
istoricul browserului, log-urile de acces ale serverului și ale proxy-ului Railway, header-ul
`Referer` către orice resursă externă din pagină, și cutiile poștale ale destinatarilor dacă linkul
e vreodată trimis prin email.

### ⭐ Vestea bună: e COD MORT

template-urile de email și în server:

```bash
# → doar ruta însăși (crud.mjs:837-838) și testele (care folosesc COOKIE, nu ?token=)
```
